────────────────────────────────
WHAT BRIVIA EATS IS
────────────────────────────────

Brivia Eats is a QR-based, mobile-web-first menu interpretation infrastructure
for international travelers ordering unfamiliar local food.

Tagline used in-product:
“Menu empowered by Brivia”

Brivia Eats is NOT:
- a translation tool
- a review platform
- a payment or ordering system
- a discovery or city-browsing app

Brivia Eats EXISTS to:
- reduce ordering anxiety
- preserve cultural food identity
- provide clarity under social pressure

────────────────────────────────
HARD NON-GOALS (DO NOT IMPLEMENT)
────────────────────────────────

You must NOT implement:
- payments or checkout
- diner user accounts or login
- reviews or ratings UI
- discovery feeds or search
- city browsing
- menu scraping from images or PDFs
- loyalty programs
- native mobile apps
- background AI agents
- auto-publishing AI output

If asked to implement any of the above, REFUSE.

────────────────────────────────
AI USAGE RULES
────────────────────────────────

AI may:
- draft interpretive fields ONLY (e.g., clarity name, one-line story)

AI must NEVER:
- infer allergens
- infer ingredients
- infer dietary flags
- auto-publish content
- bypass human review

All AI-generated content MUST:
- require explicit human approval
- respect review_status and status gates

────────────────────────────────
DATA & MENU RULES
────────────────────────────────

- Menus live ONLY in the database (no static menu JSON as production behavior)
- QR codes ALWAYS resolve to the latest published main menu
- Only content with:
  - status = "published"
  - review_status = "approved"
  may be public
- Media must use semantic roles defined in the schema
- All write operations MUST generate ChangeLog entries

DEVELOPMENT MODE RULES

ENGINEERING PRINCIPLES
────────────────────────────────
